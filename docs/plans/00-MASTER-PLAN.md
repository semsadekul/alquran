## Why the UI keeps coming out "static" or broken (root cause)

The codebase contains **two competing styling systems**:

1. **Tailwind 4 utility classes** — used in `app/layout.tsx`, `app/page.tsx` (home). These pages look fine.
2. **Semantic CSS classes** (`page-container`, `reader-page`, `surah-header-card`, `audio-player-bar-web`, `pack-card`, `settings-group`, `empty-state`, …) — used across **21 files / ~237 occurrences** (reader, settings, offline, library, hadith, search, tafsir, hifz pages). **These classes are never defined in any CSS file.** `apps/web/app/globals.css` only defines a small legacy set (`.shell`, `.card`, `.grid`, `.eyebrow`, `.lede`, `.page-enter`, `.skeleton`, verse typography, audio-less nav styles).

Result: every page built with semantic classes renders as **unstyled raw HTML** — which is exactly the "static UI / bad layout" problem reported. Previous AI agents kept generating class names that belong to a stylesheet that does not exist in this repo.

**The fix is architectural, not cosmetic:** converge on ONE styling system (Tailwind 4 + the design tokens defined in `02-DESIGN-SYSTEM.md`) and convert every page to it. See `03-RESPONSIVE-UI-FIXES.md`.

## Non-negotiable rules for the executing agent

1. **Never invent a CSS class.** If a class is not a Tailwind utility and not defined in `globals.css`, it must not appear in JSX. Run the guard script (defined in `03-RESPONSIVE-UI-FIXES.md` §7) after every page you touch.
2. **Mobile-first.** Base styles target 360 px width; enhance with `md:` (≥768) and `lg:` (≥1024). Never use fixed pixel widths on containers; use `max-w-*` + `w-full`.
3. **Touch targets ≥ 44×44 px** for every interactive element on mobile.
4. **Arabic is RTL.** Every Arabic text block gets `dir="rtl"`, `lang="ar"`, and the `font-arabic` family. Never mix Arabic into an LTR flex row without explicit direction handling.
5. **The app must build as a static export.** After each phase run: `pnpm --filter @alquran/web build` — it must succeed with `output: 'export'` (Capacitor consumes `apps/web/out`). There are **no server API routes** — never call `fetch('/api/...')`; read data at build time (server components + `generateStaticParams`) or from bundled JSON / IndexedDB / localStorage at runtime.
6. **No network-dependent assets in the shell.** Fonts, icons, CSS, JS must all be bundled (the Android app must render perfectly in airplane mode). Only Quran audio streaming/downloading may touch the network.
7. **TypeScript must pass:** `pnpm --filter @alquran/web typecheck` after every phase.
8. `apps/mobile`, `apps/api`, `apps/al_quran` are dead — **remove them in Phase 1's first commit** (see owner decision #4); until removed, do not touch them. Do not edit generated dirs: `apps/web/out`, `.next`, `android/app/src/main/assets/public` (regenerated by `cap sync`).
9. **Preserve user data:** existing IndexedDB (`AlQuranOfflineDB` v1) and localStorage keys (`alquran_preferences`, `alquran_last_read`) must keep working. Only *add* stores via DB version bump (v2), never rename existing ones.
10. **Small commits per page/feature** with the plan section number in the message, e.g. `feat(ui): convert /settings to design system [03 §4.6]`.
